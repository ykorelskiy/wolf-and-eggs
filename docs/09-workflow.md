# Workflow

Как практически работать над проектом: цикл от идеи до коммита, роли участников, что делает человек, что делает Cline, что — Nano Banana.

## Роли

### Человек (ты)

- Принимает все ключевые решения: геймплей, арт, звук, архитектурные изменения.
- Генерирует ассеты через Nano Banana и интегрирует их в репозиторий.
- Тестирует игру в браузере (десктоп + мобильный).
- Делает финальные коммиты и пуш.
- Решает, когда фаза закрыта.

### Cline (AI-агент в редакторе)

- Пишет и правит код по чётко поставленным задачам.
- Пишет и правит документы по запросу.
- Готовит промпты для Nano Banana.
- Предлагает план, diff, коммит-сообщения.
- **Не делает ничего без подтверждения.** Контракт — в `08-decisions.md`.

### Nano Banana

- Генерирует пиксель-арт и (если применимо) звуки по промптам из `docs/10-prompts/nano-banana/`.
- Запускается вне проекта, результат человек кладёт в `assets/sprites/remaster/` или `assets/audio/remaster/`.

## Основной цикл работы

Один проход цикла = одна задача из roadmap (или её часть).

```
┌─────────────────────────────────────────────┐
│  1. Выбрать задачу из 06-roadmap.md        │
│  2. Открыть релевантные документы          │
│  3. Поставить задачу Cline                  │
│  4. Cline предлагает план                   │
│  5. Согласовать план                        │
│  6. Cline пишет код / правит документы     │
│  7. Запустить локально, протестировать     │
│  8. Зафиксировать в roadmap                 │
│  9. Коммит                                  │
└─────────────────────────────────────────────┘
```

### Шаг 1. Выбор задачи

- Открываем `06-roadmap.md`.
- Берём первую невыполненную задачу из текущей фазы.
- Если задача слишком крупная — разбиваем на 2–3 подзадачи в голове (или прямо в roadmap).

### Шаг 2. Открыть документы

Минимальный набор перед началом любой кодовой задачи:
- `02-architecture.md` — что за модуль и как он связан с другими
- `07-progress-log.md` — дневник прогресса по фазам
- `08-decisions.md` — ADR, ключевые решения и их обоснование

Дополнительно по теме:
- Геймплейная задача → `01-gdd.md`
- Графика → `03-art-direction.md`, `04-asset-manifest.md`
- Звук → `05-audio-manifest.md`

### Шаг 3. Поставить задачу Cline

Формат хорошей задачи:

> «Реализуй модуль `src/systems/egg-spawner.js` из Phase 3. Спецификация в `02-architecture.md`, события — в `02-architecture.md`, раздел «События». Используй пул объектов. Спавн и скорость — из `src/config.js`.»

Плохая задача:

> «Сделай яйца».

Чем конкретнее ссылки на документы, тем меньше отсебятины.

### Шаг 4. План от Cline

Cline отвечает планом:
- какие файлы создаст / изменит;
- какие функции / классы внутри;
- какие события эмитит и слушает;
- что **не** будет делать (на случай, если задача оказалась шире).

### Шаг 5. Согласование

Читаем план. Если что-то не так — правим **до написания кода**. Если ок — даём «поехали».

### Шаг 6. Написание кода

Cline пишет код. Человек:
- следит, чтобы не было drive-by изменений в посторонних файлах;
- задаёт уточняющие вопросы, если что-то странное;
- не вмешивается без необходимости.

### Шаг 7. Локальное тестирование

Минимальный чек на каждой задаче:
- Игра запускается (`index.html` через локальный сервер).
- Нет ошибок в консоли.
- Конкретная фича работает как ожидалось.
- Старые фичи не сломались (быстрая проверка).

**Локальный сервер.** Двойной клик по `index.html` не подойдёт — ES modules требуют http(s). Варианты:

```bash
# Python 3
python3 -m http.server 8000

# Node.js (если установлен)
npx serve

# VSCode
# Live Server extension
```

### Шаг 8. Обновление roadmap

Если задача из roadmap выполнена полностью — Cline предлагает изменить `[ ]` на `[x]` в `06-roadmap.md`. Если фаза закрыта целиком — отдельно отмечаем это.

### Шаг 9. Коммит

Формат — из `07-progress-log.md`:

```
egg-spawner: добавить пул объектов и спавн по таймеру
docs: обновить roadmap, phase 3 в работе
```

Cline предлагает сообщение, человек жмёт коммит.

## Цикл работы с ассетами

Отдельный поток, идёт параллельно с кодом начиная с Phase 5.

```
┌──────────────────────────────────────────────┐
│  1. Открыть 04-asset-manifest.md            │
│  2. Выбрать ассет со статусом missing       │
│  3. Cline пишет/обновляет промпт            │
│  4. Запустить Nano Banana с промптом         │
│  5. Получить PNG, проверить                  │
│  6. Положить в assets/sprites/remaster/     │
│  7. Интегрировать в код (если нужно)        │
│  8. Обновить статус в манифесте на done     │
│  9. Коммит                                   │
└──────────────────────────────────────────────┘
```

### Структура промпта для Nano Banana

Файл: `docs/10-prompts/nano-banana/<asset-name>.md`

Содержит:
- **Объект:** что генерируем (волк в позиции TL).
- **Размер:** в пикселях, например 48×48.
- **Стиль:** pixel art, ссылка на `03-art-direction.md`.
- **Палитра:** конкретные hex-цвета или ссылка на палитру.
- **Фон:** прозрачный.
- **Анимация:** количество кадров, описание разницы между ними.
- **Референсы:** опционально — словесные.
- **Что не должно быть:** анти-промпт (без теней, без градиентов, без сглаживания).

### Структура промпта для звуков

То же самое, но:
- **Длительность** в секундах.
- **Характер:** короткий «pluck», глубокий «thud», и т.п.
- **Источник:** «8-bit chiptune», «retro arcade», «LCD beep».
- **Тональность:** если важна.
- Формат — MP3 или WAV (потом конвертируем).

Звуки Nano Banana может не генерировать — тогда используем другие AI или библиотеки free SFX, фиксируем источник в `05-audio-manifest.md`.

## Цикл работы с документами

Если в процессе работы стало понятно, что документ устарел или содержит ошибку:

1. **Не править молча.** Сначала обсудить, в чём именно несоответствие.
2. Решить: править документ под код, или код под документ.
3. Cline предлагает точечный патч документа.
4. Коммит с scope `docs:`.

Документация и код должны быть согласованы на каждом коммите в `main`.

## Тестирование между фазами

После закрытия каждой фазы — обязательная проверка:

- Десктоп: Chrome, Firefox (Safari — если есть mac).
- Мобильный: реальное устройство, не только эмулятор DevTools.
- Все клавиши и тач-зоны.
- Пауза и рестарт.
- Звук с muted=true и muted=false.
- Сохранение рекорда между перезагрузками.

Чек-лист — в `docs/testing-checklist.md` (создаётся в Phase 7).

## Когда что-то идёт не так

### Cline пишет ерунду

- Остановить.
- Указать, что именно неправильно (со ссылкой на документ).
- Попросить переписать конкретный кусок.
- Если повторяется — упростить задачу, разбить на меньшие.

### Сломалось то, что работало

- Откатить последний коммит, если он подозрительный (`git revert`).
- Если непонятно, что сломало — поставить Cline задачу: «локализуй регрессию».
- Не «фиксить наугад» — найти причину.

### Документы противоречат друг другу

- Зафиксировать конфликт письменно.
- Принять решение, какой документ — источник истины в этом вопросе.
- Обновить противоречащий документ.
- Коммит `docs: устранить противоречие между X и Y`.

### Nano Banana выдаёт не то

- Не пытаться «исправить руками» в Photoshop, если есть быстрый путь поправить промпт.
- Уточнить промпт (палитра, размер, анти-промпт).
- 3 неудачных попытки → отметить ассет как `blocked` в манифесте, перейти к следующему, вернуться позже.

## Структура рабочей сессии

Типичная сессия за компьютером:

1. `git pull` (если работаешь с разных машин).
2. Запустить локальный сервер.
3. Открыть `06-roadmap.md`, выбрать задачу.
4. Прогнать цикл «задача → Cline → код → тест → коммит».
5. Сделать 1–3 таких цикла.
6. `git push`.
7. Закрыть редактор.

Не пытаться «закрыть всю фазу за один присест», если фаза большая. Лучше 2 коммита в день, но рабочих.

## Минимальный санити-чек перед пушем в main

- [ ] Игра запускается без ошибок в консоли.
- [ ] Все ранее работавшие фичи всё ещё работают.
- [ ] `06-roadmap.md` соответствует реальности.
- [ ] Манифесты ассетов соответствуют файлам в `assets/`.
- [ ] Коммиты осмысленные, не «.» и не «wip».

## Что считается завершением MVP

Phase 7 закрыта целиком:
- Все спрайты — финальные, не плейсхолдеры.
- Все звуки — финальные.
- Игра проходится на десктопе и мобильном без багов.
- README в порядке.
- (Опционально) задеплоено.

Дальше — Phase 8 (LCD) или Phase 9 (расширения), на выбор и по настроению.
